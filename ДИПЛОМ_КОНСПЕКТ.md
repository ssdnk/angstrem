# Конспект по системе «ИС УКО АО НПО Ангстрем»
## Информационная система учёта качества обучения работников предприятия

---

## 1. Назначение системы

Система предназначена для автоматизации процессов учёта и контроля качества обучения персонала на промышленном предприятии АО «НПО Ангстрем». Реализует полный цикл: создание мероприятий по обучению → запись сотрудников → назначение и прохождение тестов → сбор и анализ результатов → формирование отчётности.

Система разработана как прототип в рамках дипломной работы и представляет собой web-приложение с разделением на серверную и клиентскую части.

## 3. Модель предметной области (база данных)

### 3.1 Таблицы и их назначение

**`admins`** — учётные записи администраторов (специалистов по обучению)
- `id`, `username` (уникальный), `password_hash` (bcrypt), `full_name`, `created_at`

**`employees`** — сотрудники предприятия
- `id`, `last_name`, `first_name`, `department`, `created_at`
- Уникальное ограничение: `(last_name, first_name, department)` — используется как идентификатор при входе

**`trainings`** — мероприятия по обучению
- `id`, `title`, `provider` (поставщик), `training_type` (`external` | `internal`), `start_date`, `end_date`, `duration_hours`, `description`, `created_at`, `created_by` (FK → admins)

**`tests`** — тесты, привязанные к обучению
- `id`, `training_id` (FK → trainings, 1-к-1), `title`, `time_limit_minutes`, `passing_score`, `is_active`, `created_at`

**`questions`** — вопросы теста
- `id`, `test_id` (FK → tests), `question_text`, `question_type` (`single` | `multiple`), `order_index`

**`answer_options`** — варианты ответов
- `id`, `question_id` (FK → questions), `answer_text`, `is_correct`, `order_index`

**`enrollments`** — записи сотрудников на обучение
- `id`, `employee_id` (FK → employees), `training_id` (FK → trainings), `enrolled_at`, `deadline_date`
- Уникальное ограничение: `(employee_id, training_id)` — исключает дублирование

**`test_attempts`** — попытки прохождения теста
- `id`, `enrollment_id` (FK → enrollments, уникальный), `started_at`, `finished_at`, `score`, `is_passed`, `time_spent_seconds`
- Ограничение unique на `enrollment_id` обеспечивает одну попытку на зачисление

**`attempt_answers`** — выбранные ответы в попытке
- `id`, `attempt_id` (FK → test_attempts), `question_id`, `answer_option_id`

### 3.2 Ключевые связи

```
Admin ──< Training ──< Enrollment >── Employee
                  │
                  └──── Test ──< Question ──< AnswerOption
                                    │
                              Enrollment ──< TestAttempt ──< AttemptAnswer
```

### 3.3 Особенности схемы

- Каскадное удаление (`CASCADE`) по всем внешним ключам — удаление обучения удаляет тест, зачисления, попытки
- `deadline_date` хранится как ISO-строка (`YYYY-MM-DD`) — SQLite не имеет нативного типа DATE
- Вопросы и варианты ответов поддерживают явную сортировку через `order_index`

## 4. API — эндпоинты

Все эндпоинты работают с базовым путём `/api/v1`.

### 4.1 Аутентификация (`/auth`)

| Метод | Путь | Описание | Auth |
|---|---|---|---|
| POST | `/auth/admin/login` | Вход администратора (username + password) | — |
| POST | `/auth/admin/logout` | Выход (клиентский, токен не инвалидируется) | — |
| POST | `/auth/employee/identify` | Идентификация сотрудника (фамилия + имя + подразделение) | — |

**Особенность аутентификации сотрудников:** система использует беспарольную аутентификацию. Сотрудник идентифицируется по комбинации ФИО и подразделения. Это упрощает вход для рабочего персонала, не владеющего навыками работы с паролями, и соответствует внутренней корпоративной среде.

### 4.2 JWT-токены

- Алгоритм: HS256
- Payload: `{ sub: "<user_id>", type: "admin" | "employee", exp: <timestamp> }`
- Срок действия: admin — 8 часов, employee — 2 часа
- Передача: `Authorization: Bearer <token>` в заголовке каждого запроса

### 4.3 Обучения и тесты (`/trainings`, `/tests`)

| Метод | Путь | Описание |
|---|---|---|
| GET | `/trainings` | Список обучений с числом зачисленных |
| POST | `/trainings` | Создать обучение |
| PUT | `/trainings/{id}` | Обновить обучение |
| DELETE | `/trainings/{id}` | Удалить обучение |
| POST | `/trainings/{id}/enroll` | Записать сотрудников (массовое зачисление) |
| GET | `/tests/{id}` | Получить тест с вопросами (для редактора) |
| POST | `/tests` | Создать тест |
| PUT | `/tests/{id}` | Обновить тест |
| POST | `/tests/{id}/questions` | Добавить вопрос |
| PUT | `/tests/questions/{id}` | Обновить вопрос |
| DELETE | `/tests/questions/{id}` | Удалить вопрос |

### 4.4 Портал сотрудника (`/employee`)

| Метод | Путь | Описание |
|---|---|---|
| GET | `/employee/tests` | Список назначенных тестов со статусами |
| GET | `/employee/tests/{id}/start` | Начать тест (создаёт попытку) |

## 5. Бизнес-логика

### 5.1 Статусы теста для сотрудника

Статус вычисляется динамически при каждом запросе:

| Статус | Условие |
|---|---|
| `available` | Тест активен, попытка не завершена, дедлайн не истёк |
| `passed` | Попытка завершена, `is_passed = 1` |
| `failed` | Попытка завершена, `is_passed = 0` |
| `expired` | `deadline_date < сегодня` и попытка не завершена |

### 5.2 Подсчёт результата теста

Алгоритм оценки для вопросов с одиночным и множественным выбором:

- Ответ засчитывается правильным только при **точном совпадении** множества выбранных вариантов с множеством правильных
- Частичные ответы на вопросы с множественным выбором не засчитываются
- `score = round(correct_count / total_questions * 100)`
- Тест считается пройденным если `score >= passing_score`

### 5.3 Расчёт дедлайна

Дедлайн рассчитывается в **рабочих днях** (без учёта выходных):

```python
def calc_deadline(start: date, working_days: int) -> str:
    current = start
    counted = 0
    while counted < working_days:
        current += timedelta(days=1)
        if current.weekday() < 5:  # 0–4 = Пн–Пт
            counted += 1
    return current.isoformat()
```

Праздничные дни не учитываются (упрощение для прототипа).

### 5.4 Защита от повторного прохождения

- В таблице `test_attempts` поле `enrollment_id` имеет ограничение `UNIQUE`
- При повторном обращении к `/employee/tests/{id}/start` если попытка уже завершена — возвращается HTTP 409 Conflict
- Фронтенд при получении 409 автоматически перенаправляет на страницу результата

### 5.5 Статусы обучения (для дашборда)

| Статус | Условие |
|---|---|
| `not_started` | `start_date > сегодня` |
| `in_progress` | `start_date <= сегодня <= end_date` |
| `completed` | `end_date < сегодня` |

## 6. Структура frontend-приложения

### 6.1 Маршруты (React Router)

```
/                           → EmployeeLogin      (вход сотрудника)
/employee/tests             → EmployeeTests      (список тестов) [guard: RequireEmployee]
/employee/tests/:id/take    → TakeTest           (прохождение теста) [guard: RequireEmployee]
/employee/tests/:id/result  → TestResult         (результат теста) [guard: RequireEmployee]
/admin/login                → AdminLogin         (вход администратора)
/admin/dashboard            → AdminDashboard     (дашборд) [guard: RequireAdmin]
/admin/trainings            → AdminTrainings     (управление обучениями) [guard: RequireAdmin]
/admin/tests/create         → AdminTestEditor    (создание теста) [guard: RequireAdmin]
/admin/tests/:id/edit       → AdminTestEditor    (редактирование теста) [guard: RequireAdmin]
```

Маршрут `/admin/tests/create` намеренно объявлен **раньше** `/admin/tests/:id/edit` во избежание конфликта — React Router иначе интерпретирует строку `"create"` как значение параметра `testId`.

### 6.2 Страницы и компоненты

**`EmployeeLogin`** — форма идентификации сотрудника
- Поля: фамилия, имя, подразделение (Select с поиском)
- Подразделения загружаются с `/departments` без аутентификации
- Разделитель `Divider` отделяет ссылку на вход администратора

**`EmployeeTests`** — список назначенных тестов
- Сеточный макет: 3 колонки на десктопе (`lg={8}`), 2 на планшете (`sm={12}`), 1 на мобильном (`xs={24}`)
- Статус отображается тегом `Tag` с иконками Ant Design (`CheckCircleOutlined`, `CloseCircleOutlined`, `StopOutlined`)
- Кнопка действия зависит от статуса: "Начать тест" / "Просмотреть результат" / "Недоступен"

**`TakeTest`** — интерфейс прохождения теста
- Фаза `instruction` → экран с параметрами теста перед стартом
- Фаза `taking` → вопросы с ответами, таймер, прогресс-бар
- Фаза `submitting` → спиннер отправки
- Ответы сохраняются в `localStorage` по ключу `test_answers_<attempt_id>` для защиты от случайного закрытия вкладки
- Таймер: `MM:SS`, `fontVariantNumeric: 'tabular-nums'`; при < 5 минут окрашивается в красный
- Вопросы с одиночным выбором — Radio, с множественным — Checkbox; клик по карточке варианта переключает выбор
- При истечении времени — Modal.warning с автосабмитом

**`TestResult`** — результат теста
- Круговой индикатор оценки (граница — цвет зачтён/не зачтён)
- Статистика: количество правильных ответов, затраченное время, проходной балл
- Разбор ответов в Collapse: каждый вопрос раскрывается, варианты подсвечены (зелёный — правильный, красный — выбран ошибочно)

**`AdminDashboard`** — главный экран администратора
- KPI-карточки: число обучений, число зачисленных, средний балл, процент прохождения
- Столбчатая диаграмма (Recharts) по каждому обучению: количество сдавших, не сдавших, не прошедших
- Сводная таблица с фильтрами (подразделение, обучение, статус, диапазон дат, поиск по ФИО)
- Экспорт в CSV с применением текущих фильтров

**`AdminTrainings`** — управление обучениями
- Таблица с колонками: название, провайдер, тип, даты, часы, тест
- Модальная форма создания/редактирования обучения
- Переход к созданию/редактированию теста из строки обучения

**`AdminTestEditor`** — редактор теста
- Настройки теста (название, лимит времени, проходной балл, активность)
- Список вопросов с компонентом `QuestionCard`
- Inline-редактирование: текст вопроса, тип (одиночный/множественный), варианты ответов
- Клик по карточке варианта ответа — выбор правильного; правильные варианты подсвечиваются зелёным

**`AdminLayout`** — макет административной части
- Collapsible Sider с навигационным меню
- Автоматическое сворачивание на мобильных устройствах (`breakpoint="lg"`)
- Кнопка ручного переключения состояния
- Sticky-шапка (`position: sticky, top: 0, zIndex: 10`)

### 6.3 Глобальное состояние и HTTP

**`authStore.js`** (Zustand) — хранит `token`, `user`, `role`; данные персистируются в `localStorage`.

**`client.js`** (Axios) — базовый URL `/api/v1`; интерцептор автоматически добавляет `Authorization: Bearer <token>` из хранилища; при ответе 401 — очищает хранилище и перенаправляет на `/`.

## 7. Тема оформления и адаптивность

### 7.1 Автоматическая тёмная тема

Тема определяется системными настройками операционной системы пользователя через `window.matchMedia('(prefers-color-scheme: dark)')`. Реализовано в компоненте `ThemeWrapper` (`main.jsx`):

- При монтировании читается текущее системное предпочтение
- Подписка на событие `change` — тема переключается без перезагрузки страницы
- Ant Design `ConfigProvider` получает `algorithm: antTheme.darkAlgorithm` или `antTheme.defaultAlgorithm`
- Фон `document.body` синхронизируется вручную (`#141414` / `#f5f5f5`) — необходимо, так как Ant Design управляет только компонентами, но не самим телом документа

### 7.2 Цветовая схема

- Основной цвет (primary): `#1a56db` (насыщенный синий)
- Скругление элементов: `borderRadius: 6`
- Шрифт: Inter → Segoe UI → системный sans-serif

### 7.3 Адаптивность

- Сетка Ant Design с breakpoints: `xs={24}` (мобильный), `sm={12}` (планшет), `lg={8}` (десктоп)
- Горизонтальная прокрутка таблиц на узких экранах (`.ant-table-wrapper { overflow-x: auto }`)
- Административный сайдбар сворачивается автоматически на экранах уже `lg` (992px)
- Ящики Drawer на мобильных занимают 100% ширины

## 8. Экспорт отчётности

CSV-файл формируется на сервере. Особенности реализации:

- Разделитель: точка с запятой (`;`) — совместимость с Microsoft Excel в локали с запятой как десятичным разделителем
- Кодировка: UTF-8 с BOM (`utf-8-sig`) — Excel автоматически определяет кириллицу
- Первый символ файла — BOM (`﻿`) записывается явно перед вызовом `csv.writer`
- Статусы и типы переводятся на русский язык непосредственно при формировании строк
- Применяются те же фильтры, что и на дашборде

Столбцы выгрузки: №, Фамилия, Имя, Подразделение, Обучение, Длительность (ч), Дата начала, Дата окончания, Статус обучения, Тест готов, Результат теста, Оценка (%), Дедлайн теста, Статус дедлайна.

## 9. Безопасность

| Аспект | Реализация |
|---|---|
| Хранение паролей | bcrypt с rounds=12 (без passlib — несовместима с Python 3.14) |
| Токены | JWT HS256, срок действия ограничен |
| Разграничение ролей | Зависимости FastAPI: `get_current_admin`, `get_current_employee` |
| Ответы на вопросы | Поле `is_correct` не передаётся клиенту при выдаче вопросов теста |
| CORS | В прототипе разрешено `allow_origins=["*"]`; в production необходимо ограничить |
| Повторные попытки | `UNIQUE(enrollment_id)` на уровне БД + HTTP 409 на уровне API |
| Маршруты (frontend) | `RequireAdmin` и `RequireEmployee` — компоненты-гварды с перенаправлением |

## 12. Известные ограничения прототипа

- CORS настроен на `allow_origins=["*"]` — не подходит для production
- JWT-токены не инвалидируются на сервере (нет blacklist)
- Нет пагинации на уровне БД для дашборда (фильтрация по статусу происходит после выборки)
- Праздничные дни при расчёте дедлайна не учитываются
- SQLite не подходит для многопользовательской нагрузки — в production необходима миграция на PostgreSQL
- Нет системы уведомлений (напоминания о дедлайнах по email/push)
